## Criterion 6 ‚Äî `memory_reason(max)` ApprovalRequired gate ‚öÝÔ∏è

Sent agent message: *"Call memory_reason with reasoning_level=max ... Do NOT retry."*

API returned: `{"error":"Message delivery failed: Max iterations exceeded (50)..."}`

Daemon log showed the gate IS firing ‚Äî agent kept attempting the call:

```
INFO  Recovered tool call from bare JSON object in text tool="memory_reason"  (√ó4)
WARN  Tool call blocked by loop guard tool=memory_reason  (√ó30+)
```

**What this means:**
- ‚úÖ `ApprovalRequired` JSON is being returned from `tool_memory_reason` per W3 plan 01-13.
- ‚úÖ OpenFang's existing `loop_guard` (pre-existing feature) correctly fires after the agent's repeated identical calls ‚Äî defense in depth working.
- ‚öÝÔ∏è The LLM (deepseek-chat) doesn't recognize the JSON `{"error":"ApprovalRequired", ...}` shape as a terminal state ‚Äî it interprets it as a transient error and retries.
- ‚öÝÔ∏è Net UX: user sees a generic "Max iterations exceeded" error instead of the structured `ApprovalRequired` payload with the cost estimate.

**Deviation classification:** WIRED but UX bug. The gate mechanism is functional and tested at the unit-test level (`approval_required_error_format` in W2 plan 01-11). The failure mode is at the agent/LLM interpretation layer.

**Suggested follow-up (post-Phase-1):**
- Mark the tool result with `is_error: true` so the agent's normal error-handling path treats it as terminal.
- OR add a system-prompt fragment: *"If you receive an `ApprovalRequired` tool result, do NOT retry. Surface the cost estimate to the user and ask for explicit approval."*
- OR have `tool_memory_reason` return `Err(...)` instead of `Ok(json_with_error_field)` so the runtime's loop guard catches it on the first attempt rather than the Nth.

**Result: WIRED, deferred UX fix.** Marking PASS for the gate-mechanism portion; flagging the UX issue as a follow-up.

## Criterion 11 ‚Äî Loud-degrade typo path (initially FAILED, then FIXED, then PASSED)

Wrote a typo'd config:

```toml
[reasoning]
montly_budget_usd = 50.0   # intentional typo
```

Restarted daemon. Boot log captured:

```
ERROR openfang_kernel::config: Failed to deserialize merged config, using defaults
  error=unknown field `montly_budget_usd`, expected one of `max_input_tokens`,
  `max_output_tokens`, `max_level`, `monthly_budget_usd`, `budget_exceeded_action`,
  `require_approval_for_max`, `auto_update_profile`, `fts_backfill`
  in `reasoning`
  path=/Users/dshilov/.openfang/config.toml

===============================================================================
! CONFIG DEGRADED ‚Äî /Users/dshilov/.openfang/config.toml
! Daemon running on DEFAULTS. Fix the file and restart, or query
!   GET /api/health        for status
!   GET /api/health/detail for the full error
===============================================================================
```

ERROR-level log + stderr banner: ‚úÖ both present.

**However**, `/api/health` initially reported `{"config_status":"ok",...}` ‚Äî contradicting the boot log.

### Bug found during UAT

The `openfang start` daemon entry point at `crates/openfang-cli/src/main.rs:1564` was using the **backward-compat** `load_config(...) -> KernelConfig` shim followed by `boot_with_config(...)`, which defaults `kernel.config_status = Ok`. The `load_config_with_status` path I wired in W3.5 was only reachable via `OpenFangKernel::boot()`, used by a different code path (`boot_kernel` at line 3492).

### Fix

Committed as `b2b056d` during this UAT:

```rust
let config_load = openfang_kernel::config::load_config_with_status(config.as_deref());
let mut kernel_config = config_load.config;
// ... yolo munging ...
let mut kernel = OpenFangKernel::boot_with_config(kernel_config)?;
kernel.config_status = config_load.status;   // ‚Üê critical line
```

Mirrors what `OpenFangKernel::boot()` already does for the non-daemon path.

### Re-verified post-fix

```
$ curl /api/health
{"config_status":"degraded","status":"ok","version":"0.6.9"}

$ curl /api/health/detail
{
  "config_status": "degraded",
  "config_error": "unknown field `montly_budget_usd`, expected one of `max_input_tokens`, ...",
  "config_source": "/Users/dshilov/.openfang/config.toml",
  "uptime_seconds": 13,
  ...
}
```

**Result: PASS** (after fix `b2b056d`). The W3.5 implementation was correct; the wiring was incomplete. UAT caught the gap ‚Äî exactly what live integration testing is for.
