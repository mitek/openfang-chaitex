## Decisions Made

- `detect_recovery()` extracted as standalone pure function (`prev_had_error && !this_has_error`) so all three behavioral test cases exercise the real implementation directly without a full async loop harness
- Recovery weighting of 0.40 (vs 0.45 for iterations) — error-then-success is the dominant distillation signal but multi-step iteration depth remains the primary proxy for "non-trivial procedure"
- `error_recovery_count: 0` at all sites that exit before the ToolUse path (silent, MaxContinuations, WASM executor, Python executor, daemon-fallback) — these exits precede or bypass tool iterations, so recovery tracking does not apply
