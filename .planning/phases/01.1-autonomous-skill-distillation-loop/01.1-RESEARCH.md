# Phase 1.1: Autonomous Skill Distillation Loop — Research

**Researched:** 2026-06-10
**Domain:** Rust async agent loop wiring, background tokio tasks, SkillRegistry draft state, ReasoningEngine invocation from kernel context
**Confidence:** HIGH (all findings sourced from direct codebase reads)

---

## Summary

Phase 1 shipped every load-bearing mechanism — `skill_manage`, FTS5 session search, `memory_reason`/`memory_conclude`, and `BudgetTracker` — but nothing fires them automatically. Phase 1.1 adds three autonomous wiring layers on top: a post-turn reflection hook inside `execute_llm_agent` that scores completed turns and optionally enqueues a distillation job; a failure-then-recovery counter attached to `skill_execute` results that accumulates until a patch proposal threshold is reached; and a kernel-internal background tokio task (modeled after the existing `memory.consolidate()` loop) that runs `memory_conclude` over recent FTS5-indexed sessions on a configurable schedule.

The codebase pattern for kernel-internal background work is established and clear: `tokio::spawn` inside `start_background_tasks` at `kernel.rs:4671`, with an interval loop, shutdown check via `self.supervisor.is_shutting_down()`, and an `Arc::clone(self)` capture. This is the correct home for the memory-consolidation nudge.

The correct hook point for post-turn reflection is the block at `kernel.rs:3033–3088`, immediately after `run_agent_loop` returns. At that point the kernel holds `result.iterations` (LLM round-trips = proxy for tool-call count), `result.total_usage` (tokens), and the elapsed wall time since the start of `execute_llm_agent`. Error-recovery events are not yet counted at the loop level; they need either a new `AgentLoopResult` field or a separate turn-stats struct.

The "Continuous/Proactive" wording in the ROADMAP refers to `ScheduleMode::Continuous` and `ScheduleMode::Proactive` in `openfang-types::agent::ScheduleMode` — these are schedule modes, not `AgentMode`. The actual `AgentMode` enum has three variants: `Observe`, `Assist`, `Full`. The draft-skill safety rail should key off `capabilities.allow_skill_mutation` (already exists, default false) rather than trying to read `ScheduleMode`.

Skills already have an `enabled: bool` field on `InstalledSkill`. Setting `enabled = false` at create time is the natural draft mechanism — no new struct or directory needed. The approval path can reuse the existing `ApprovalManager` (kernel.approval_manager) which already has pending/recent lists surfaced on the dashboard `/api/approvals` route.

**Primary recommendation:** Model all three components as purely additive wiring layers over existing mechanisms. No new crates needed. Add a `TurnStats` struct to `openfang-runtime` to carry per-turn reflection data out of the agent loop, implement the distillation job queue as a `VecDeque<DistillationJob>` on the kernel (bounded, async-friendly), and register the memory-consolidation nudge background task in `start_background_tasks` next to the existing consolidation loop.

## Architecture Patterns

### Pattern 1: Kernel-Internal Background Task (confirmed precedent)

The existing memory consolidation loop at `kernel.rs:4671` is the exact pattern for the memory-consolidation nudge:

```rust
// Source: crates/openfang-kernel/src/kernel.rs:4671
{
    let interval_hours = self.config.memory.consolidation_interval_hours;
    if interval_hours > 0 {
        let kernel = Arc::clone(self);
        tokio::spawn(async move {
            let mut interval = tokio::time::interval(std::time::Duration::from_secs(
                interval_hours * 3600,
            ));
            interval.tick().await; // Skip first immediate tick
            loop {
                interval.tick().await;
                if kernel.supervisor.is_shutting_down() {
                    break;
                }
                match kernel.memory.consolidate().await { ... }
            }
        });
    }
}
```

**Use this pattern verbatim for MC-01.** Place it immediately after the existing consolidation block in `start_background_tasks`.

### Pattern 2: Post-Turn Hook Point (kernel.rs execute_llm_agent)

The post-turn hook attaches at `kernel.rs:3033`, after `run_agent_loop` returns and before `metering.record`:

```rust
// Proposed insertion point — kernel.rs after line 3031:
let turn_stats = TurnStats {
    iterations: result.iterations,
    total_tokens: result.total_usage.total(),
    wall_ms: turn_start.elapsed().as_millis() as u64,
    error_recovery_count: result.error_recovery_count, // new field on AgentLoopResult
};
if let Some(ref distillation_queue) = self.distillation_queue {
    maybe_enqueue_distillation(agent_id, session_id, &turn_stats, distillation_queue,
        &self.config.distillation);
}
```

The `turn_start` instant must be captured at the top of `execute_llm_agent` before the session load.

### Pattern 3: AgentLoopResult Extension

`AgentLoopResult` needs one new field to carry error-recovery count out of the loop. The field is zero-cost when unused:

```rust
// Source: crates/openfang-runtime/src/agent_loop.rs:249 (existing struct)
pub struct AgentLoopResult {
    pub response: String,
    pub total_usage: TokenUsage,
    pub iterations: u32,
    pub cost_usd: Option<f64>,
    pub silent: bool,
    pub directives: ReplyDirectives,
    // NEW for Phase 1.1:
    pub error_recovery_count: u32,  // incremented when is_error turns to non-error
}
```

The agent_loop already tracks `error_count` per-iteration (line 1067). Adding a `bool` toggle to detect `was_error_last_tool` → `now_not_error` gives failure-then-recovery events.

### Pattern 4: Draft Skills via enabled=false

Skills already have `InstalledSkill { manifest, path, enabled: bool }`. Create distilled skills with `enabled = false`; expose them via `list_all()` (dashboard accessor). The existing `set_skill_enabled(name, true)` mutation method is the approval action.

No new struct needed. The approval flow:
1. `create_skill(name, toml, prompt_context, None)` — creates on disk
2. Immediately call `set_skill_enabled(name, false)` — marks as draft
3. Surface via `/api/distillation/drafts` (reads `list_all()`, filters `enabled=false`)
4. Approval: call `set_skill_enabled(name, true)` + audit append

**Alternative (simpler):** Pass `enabled=false` directly into `InstalledSkill` at creation time by adding an `enabled` parameter to `create_skill`. This avoids the two-step race condition.

### Pattern 5: SkillFailureTracker

Model as a `DashMap<(String, AgentId), VecDeque<FailureEvent>>` where `FailureEvent = { timestamp: Instant, error_hash: u64 }`. Lives on the kernel struct alongside `cron_scheduler`. Bounded per key (max 20 events). Rolling window: drop entries older than configurable TTL (default 7 days).

```rust
// Placed on OpenFangKernel alongside approval_manager

```

### Anti-Patterns to Avoid

- **Calling ReasoningEngine from inside the agent loop:** The loop is already async and holding session state; scheduling a background job and exiting is correct. The distillation worker runs asynchronously after the turn completes.
- **Adding keys to `[reasoning]` config section:** It uses `deny_unknown_fields`. Add a new `[distillation]` top-level section in `KernelConfig`.
- **Using AgentMode for draft-skill gating:** `AgentMode` is `Observe/Assist/Full`. The correct gate is `capabilities.allow_skill_mutation`. For auto-approve (no draft): check `config.distillation.auto_approve_non_protected`.
- **Calling `memory_conclude` as a tool from within the background task:** Invoke `ReasoningEngine::reason` directly (or the `update_user_profile` / `conclude_session` helper functions from `openfang_reasoning::profile`), not via the tool dispatch path. The background task has direct kernel access.
- **Registering distillation cron jobs in `cron_jobs.json`:** System-internal loops belong in `start_background_tasks` as `tokio::spawn`, not as user-visible cron entries. Agent-facing cron tools create user jobs; internal loops live in code.
- **FTS5 dedupe for skill names:** Skills are NOT FTS5-indexed (only `session_messages` is). Dedupe by checking `skill_registry.get(candidate_name)` (name collision) and optionally computing cosine similarity on description embeddings if the embedding driver is available. Do not expect FTS5 to work for skill content.

## Common Pitfalls

### Pitfall 1: [reasoning] deny_unknown_fields Trap
**What goes wrong:** Adding `[distillation]` keys inside `[reasoning]` in config.toml causes loud-degrade failure (daemon logs ERROR, falls back to defaults, `/api/health` shows `degraded`).
**Why it happens:** `[reasoning]` uses `#[serde(deny_unknown_fields)]` per MR-05 / STATE.md.
**How to avoid:** Add a completely separate `[distillation]` table in `KernelConfig`. The `[capabilities]` section is the precedent (added as `KernelCapabilities` sub-struct during W3).
**Warning signs:** `cargo build` passes; live test shows `config_status: "degraded"` on `/api/health`.

### Pitfall 2: Holding KernelHandle Across .await in Distillation Worker
**What goes wrong:** The distillation worker calls `ReasoningEngine::reason(...)` (async) while holding an `RwLockReadGuard` on the skill registry — the guard is `!Send`, build fails.
**Why it happens:** `SkillRegistry::snapshot()` exists precisely to avoid this. Call `snapshot()` before the `.await`, release the guard, use the snapshot.
**How to avoid:** Always use `SkillRegistry::snapshot()` before any async call involving the registry. Established pattern from agent_loop.rs plan 01-09.

### Pitfall 3: Distillation Worker Races with Agent Turn for Same Session
**What goes wrong:** The distillation worker reads session messages via FTS5 while `save_session_async` is still in flight for the same session. The dual-write is inside a transaction (FTS-03), so partial writes are safe, but the worker may not see the latest messages if it reads before the commit.
**How to avoid:** The distillation job should be enqueued AFTER `save_session_async` returns (it's called before `run_agent_loop` returns). Since the hook point is after `run_agent_loop` completes, the session is already saved. No additional ordering needed.

### Pitfall 4: `create_skill` Duplicate Name on Re-distillation
**What goes wrong:** Distilling the same procedure twice calls `create_skill` with an identical name; `SkillRegistry::create_skill` returns `SkillError::AlreadyInstalled`. The worker crashes or logs an error.
**How to avoid:** Check `skill_registry.get(candidate_name).is_some()` before calling `create_skill`. If exists: either skip, or propose a `patch_skill` to update the draft. Name the distilled skill with a deterministic hash of the session_id + procedure fingerprint so re-runs produce the same name.

### Pitfall 5: SkillFailureTracker Growing Unbounded
**What goes wrong:** If a skill fails frequently, `VecDeque` entries accumulate without bound; on pc162 (1.9 GB ARM) this becomes a memory concern.
**Why it happens:** No eviction without explicit bounding.
**How to avoid:** Bound per-key at 20 entries (configurable). Drop entries older than 7 days on each insert. The tracker should be a lightweight struct with these invariants enforced in `record_failure`.

### Pitfall 6: ReasoningEngine::reason Without LLM for Minimal/Low
**What goes wrong:** `ReasoningEngine::reason(query, Minimal)` or `Low` works without an LLM wired. `Medium+` requires LLM. If the kernel's `init_reasoning_subsystem` has not been called (test context, or `set_self_handle` not called), the engine has no LLM and returns `ReasoningError::NoLlm`.
**How to avoid:** The distillation worker should call `kernel.reasoning_engine()` and handle `None` gracefully (log WARN, skip distillation). In tests, use a `MockLlm` impl of `ReasoningLlm`.

### Pitfall 7: Background Task Starting Before set_self_handle
**What goes wrong:** `start_background_tasks` is called during `boot_with_config` but `set_self_handle` is called after Arc wrapping (after boot). If the distillation worker tries to call `reasoning_engine()` immediately, the `OnceLock` is empty and returns `None`.
**How to avoid:** The `interval.tick().await` (skip first tick) pattern from the memory consolidation loop handles this naturally — the first iteration fires after one interval, by which time `set_self_handle` has been called. Confirm the distillation worker uses `interval.tick().await` before first attempt.

## Code Examples

### Invoking ReasoningEngine from a Background Kernel Task

```rust
// Source: crates/openfang-kernel/src/kernel.rs:8018 + crates/openfang-reasoning/src/lib.rs
// Background task invoking reasoning — correct pattern:
async fn run_distillation_job(kernel: &Arc<OpenFangKernel>, job: DistillationJob) {
    let engine = match kernel.reasoning_engine() {
        Some(e) => e,
        None => {
            warn!("Reasoning engine not initialized — skipping distillation");
            return;
        }
    };
    let query = format!(
        "Review session {}. Did the agent discover a reusable procedure? \
         If yes, describe it as a named skill with description and steps.",
        job.session_id
    );
    let req = openfang_reasoning::ReasoningQuery {
        query: query.clone(),
        level: openfang_reasoning::ReasoningLevel::Medium,
        caller_agent_id: Some(job.agent_id.to_string()),
        max_facts: Some(20),
    };
    match engine.reason(req).await {
        Ok(result) if result.confidence > 0.7 => {
            // Attempt skill creation as draft
        }
        Ok(result) => {
            debug!(confidence = result.confidence, "Distillation below threshold — skipped");
        }
        Err(e) => {
            warn!("Distillation reasoning failed: {e}");
        }
    }
}
```

## Key Codebase Findings

### Agent Loop Turn Data Available for Reflection Scorer

At `kernel.rs:3033` (post-`run_agent_loop`), the kernel has:
- `result.iterations: u32` — number of LLM round-trips (proxy for tool-call count; iterations - 1 = tool-call iterations)
- `result.total_usage: TokenUsage` — input + output tokens
- `result.silent: bool` — whether the agent chose not to reply
- `turn_start: Instant` — must be added before session load in `execute_llm_agent`

**Not yet available (need new field):** error-recovery event count. Currently `error_count` is computed inside the loop per-iteration but discarded. Add `pub error_recovery_count: u32` to `AgentLoopResult` — increment when `was_error` → `not_error` across consecutive tool calls.

### AgentMode vs ScheduleMode — Critical Reconciliation

The ROADMAP mentions "Continuous/Proactive" as the condition for auto-approving distilled skills. These are `ScheduleMode` variants in `openfang-types::agent::ScheduleMode`:
```rust
pub enum ScheduleMode { Reactive, Periodic { cron }, Proactive { conditions }, Continuous { check_interval_secs } }
```
But the correct gate for skill mutation capability is `KernelCapabilities.allow_skill_mutation` (default false). The planner should implement the auto-approve gate as: `config.distillation.auto_approve_non_protected` (configurable bool, default false) rather than checking `ScheduleMode`. A user can enable auto-approve explicitly in config; using schedule mode would silently auto-approve for any agent running on a cron.

### CronScheduler: NOT the Right Mechanism for Internal Loops

`crates/openfang-kernel/src/cron.rs` manages user-facing cron jobs persisted to `cron_jobs.json` with per-agent quotas, max_total_jobs limits, and an agent_id field. System-internal loops must NOT be registered here — they would count against per-agent job limits and would appear in the user's cron job list. Use raw `tokio::spawn` + `tokio::time::interval` in `start_background_tasks`, same as the memory consolidation at line 4671.

### ApprovalManager: Ready to Reuse for Draft-Skill Approval

`kernel.approval_manager` is an `ApprovalManager` that:
- Stores pending requests in a map with `oneshot` channels
- Has `list_pending()` and `list_recent(limit)` methods
- Is surfaced on `GET /api/approvals` and `POST /api/approvals/{id}/approve|reject`
- Has a timeout mechanism (default 300s after which `ApprovalDecision::TimedOut`)

Draft-skill approval can create an `ApprovalRequest { agent_id, tool_name: "skill_distillation_approve", description: "Approve draft skill: {name}", risk_level: RiskLevel::Low }` and wait for the human to approve via dashboard. The existing approval UI tab handles this without code changes to the frontend.

### FTS5 Dedupe — Skills Are NOT Indexed

The FTS5 index covers only `session_messages` content. There is no FTS5 index over skill names, descriptions, or prompt content. Deduplication of candidate distilled skills must use:
1. Name collision check: `registry.get(candidate_name).is_some()`
2. Description similarity: string distance (simple Levenshtein on descriptions, no LLM needed) or embedding cosine similarity if `kernel.embedding_driver` is available
3. Prompt content substring check: `registry.list_all().iter().any(|s| content_overlap(&s.manifest.prompt_context, &candidate_content))`

### SkillRegistry.list() vs list_all()

- `list()` — only `enabled=true` skills — this is the agent's view
- `list_all()` — all skills including disabled — draft skills (enabled=false) will only appear here

Dashboard must call `list_all()` to surface pending drafts. The existing dashboard "Skills" tab likely calls `list()` via the API — verify and adjust the `/api/distillation/drafts` endpoint to call `list_all()` and filter `enabled=false`.

### Kernel execute_llm_agent — Where the Hook Lives

The non-streaming LLM agent path (`kernel.rs:2999`) calls `run_agent_loop` and returns `AgentLoopResult`. The hook must go in `execute_llm_agent` (the private method that both `send_message_with_handle_and_blocks` and `send_message_streaming` delegate to), not at the `send_message` layer. This ensures the hook fires for every LLM turn regardless of how the message was submitted.

## Open Questions

1. **Daily cap persistence strategy**
   - What we know: BudgetTracker uses SQLite for persistence; cron.rs uses JSON file
   - What's unclear: V10 schema migration vs lightweight JSON file — which is lower friction for the planner?
   - Recommendation: Use a JSON sidecar file (`~/.openfang/distillation_state.json`) with atomic write to avoid a schema migration. Structure: `{ "date": "2026-06-10", "distillations_today": 3 }`. Reset when date changes on read.

2. **Distillation job queue storage**
   - What we know: Jobs need to survive a short window between turn completion and worker processing; NOT needed across restarts (stale distillation jobs from days ago are useless)
   - Recommendation: In-memory `Arc<Mutex<VecDeque<DistillationJob>>>` bounded at 50 entries; no persistence needed. If the daemon restarts, the next sufficiently-complex turn re-enqueues.

3. **How to call memory_conclude for MC-01**
   - What we know: `memory_conclude` is a tool dispatch function in `tool_runner.rs` that calls `openfang_reasoning::profile::conclude_session_helper` (verify exact function name in tool_runner.rs before planning)
   - What's unclear: Can the background task call the profile helpers directly (bypassing tool dispatch), or must it go through tool dispatch?
   - Recommendation: Call the helper functions directly from `openfang_reasoning::profile` — same pattern as `memory_reason` calling `ReasoningEngine::reason` directly rather than going through `tool_memory_reason`. Check the exact helper function signatures in `crates/openfang-reasoning/src/profile.rs` before planning.

4. **SkillFailureTracker placement**
   - What we know: Needs to be kernel-global (accessed from tool dispatch for recording, and from background worker for checking thresholds)
   - Recommendation: Add `skill_failure_tracker: crate::skill_failure::SkillFailureTracker` as a field on `OpenFangKernel`. Wire it into the `KernelHandle` trait as a new default-`None` accessor so `tool_runner.rs` can record failures without knowing about the concrete kernel type.
