# Phase 01.1 Plan 04: SkillRegistry Draft Lifecycle + Dedupe Heuristic Summary

**Pure Jaccard token-set dedupe gate (SD-06) and draft create/approve/list lifecycle (SD-05) composed from existing SkillRegistry primitives, with 8 new TDD-driven tests.**

## Accomplishments

- Added `description_similarity(a, b) -> f32` ‚Äî pure Jaccard over lowercased alphanumeric word sets, no dependency additions, no LLM
- Added `is_duplicate_candidate(name, desc) -> bool` ‚Äî short-circuits on exact name match, then checks all existing skills at >= 0.85 threshold (SD-06)
- Added `create_draft_skill(name, toml, ctx) -> Result<bool, SkillError>` ‚Äî composes `create_skill` + `set_skill_enabled(false)`, returns `Ok(false)` on duplicate name instead of propagating `AlreadyInstalled` (SD-05 / Pitfall 4)
- Added `approve_draft_skill(name) -> Result<(), SkillError>` ‚Äî thin wrapper for `set_skill_enabled(name, true)` (SD-05)
- Added `list_drafts() -> Vec<&InstalledSkill>` ‚Äî filters `list_all()` to `!enabled` (SD-05)
- 8 TDD tests covering: exact-name dedupe, similar-description dedupe, distinct candidate, empty registry, draft creation+disabled state, approval+enabled state, duplicate-name skip, draft/enabled filter

## Files Created/Modified

## Decisions Made

- Jaccard token-set chosen over Levenshtein: O(n) HashSet construction vs O(mn) DP table, sufficient for skill description lengths
- Threshold 0.85 catches trailing punctuation and minor rewordings ("Deploys the app to staging" vs "Deploys the app to staging.") while leaving semantically distinct candidates unblocked
- `create_draft_skill` returns `Ok(bool)` so callers don't need to pattern-match error variants; `Ok(false)` = skip is the signal for duplicates
- All four new methods are added to the _existing_ `impl SkillRegistry` block (before the test module) ‚Äî no new impl block required
